2) Chiarisci cosa rappresenta davvero sessionToken

Tu lo chiami Bearer token, ma dici anche “sessioni server-side (PostgreSQL)” e “il backend distrugge la sessione associata al bearer token”. 

MOBILE-APP-DEVELOPER-GUIDE


Serve 2 righe esplicite per evitare implementazioni sbagliate:

è un opaque token che punta a una sessione server-side?

oppure un JWT?

scade dopo 7 giorni (lo dici) 

MOBILE-APP-DEVELOPER-GUIDE

 ma: si rinnova? c’è refresh? serve re-login?

👉 Se non c’è refresh, scriverei il comportamento atteso:

alla 401: redirect a login + wipe storage token

se possibile: “me” all’avvio per validare token

3) Workflow “startup app” (cold start) in 6 step

Manca una mini-sezione “App lifecycle” super pratica, tipo:

Leggi token da SecureStore

GET /api/auth/me

Se ok → carica feature flags + pacchetto

Setup push listeners

Registra push token se permesso concesso

Setup deep linking mapping

Questo taglia il 50% delle ambiguità.

4) Specifica paginazione, sorting, filtri (soprattutto per messages/notifications/lessons)

Hai le liste endpoint, ma non dici:

paginazione (limit/offset? cursor?)

ordinamento default

filtri supportati (es. /api/lessons?from=&to=&status= ecc.)
Se non esiste, ok, ma va dichiarato.

7) WebSocket auth: “cookie/bearer” è ambiguo

Scrivi: “Il WebSocket usa lo stesso cookie/bearer di sessione per autenticare.” 

MOBILE-APP-DEVELOPER-GUIDE


In mobile, di solito cookie non c’è. Serve una regola precisa:

token in querystring? (sconsigliato)

token in header Authorization durante handshake? (dipende dalla lib)

oppure primo messaggio WS contiene token?

Se Ampcode implementa male, le call live falliscono.
